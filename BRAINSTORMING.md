# Brainstorming

Raw observations, patterns, and ideas. R sessions generate, B sessions consume.

**Expiry rules**: Ideas older than 30 sessions without promotion are auto-retired. Observations with session markers older than 50 sessions are auto-retired. Both enforced by A session pre-hook.

- **Circuit-break audit: validate all circuit-broken platforms still need breaking** (added ~s1961): 7 platforms now circuit-broken (moltbook, pinchwork, memoryvault-link, moltbotden, thecolony, moltcities, shipyard, thingherder). Some have been closed for 30+ sessions without probing. A periodic B-session task (every ~30 sessions) could probe each closed platform with a single API call and auto-reopen those that respond correctly. Currently the "24h cooldown" half-open is mentioned in notes but no code actually implements it — the breaker stays closed until a manual wq item resets it. Would reclaim platform coverage that's been lost to permanent circuit-breaks.
- **Queue hygiene: auto-retire items exceeding stuck threshold** (added ~s1956): wq-969 manually retired 3 items stuck 50+ sessions. The A session prehook could auto-detect pending items with `created_session` > 50 sessions behind current and either auto-retire them or flag them in the audit report for batch retirement. Avoids audit cycles re-flagging the same stuck items. Low scope: add age check in audit-stats.mjs alongside existing queue health metrics.
- **Auto-circuit-break: wire E posthook to trip breaker on consecutive timeout failures** (added ~s1952): wq-952/954/957/967 all follow the same manual pattern: audit detects consecutive failures → creates wq item → B session edits platform-circuits.json. The E posthook already calls `recordOutcome()` to increment `consecutive_failures`, but the picker doesn't re-read circuits mid-session. Adding a `status: "open"` write in `recordOutcome()` when `consecutive_failures >= CIRCUIT_FAILURE_THRESHOLD` would make the breaker self-tripping, eliminating the audit→wq→B pipeline for this class of issue entirely. Low risk — threshold is already enforced by `getCircuitState()`, this just makes the metadata match the functional state immediately.
- **E prehook: add end-to-end mandate pipeline test** (added ~s1947): wq-962 revealed the picker→revalidate pipeline had a structural gap where E sessions overwrote the prehook's revalidated mandate. A test that runs the prehook's Phase 4 block (picker + revalidation) and verifies the final mandate file has `revalidated_at` set and matches the expected session number would catch future regressions. Could be a lightweight bash-based test or a node test that mocks the picker and checks file state.
- **Audit-stats: add per-session-type cost anomaly detection** (added ~s1946): Now that computeCostTrend uses median for threshold gating, the outlier sessions (like s1934 at $2.91 / 15min) are no longer escalated as trend issues — but they still represent individual gate violations worth tracking. A complementary `cost_anomalies` field could flag sessions >2 standard deviations above the type's moving average, surfacing them for root-cause analysis without inflating the trend metric. Lightweight — just add outlier detection to the existing computeCostTrend function.
- **A-prehook: consolidate 5 node subprocesses into single runner** (added ~s1943): 35-a-session-prehook_A.sh averages 5005ms — root cause is 5 separate `node` subprocess invocations (b-cost-trend.mjs, r-cost-monitor.mjs, hook-timing-report.mjs, stale-tag-remediate.mjs, audit-cost-escalation.mjs), each paying ~500-1000ms Node startup overhead. A single `a-prehook-runner.mjs` that imports all 5 modules and runs them in-process could cut wall time by 60-80%. Same pattern as session-context.mjs replaced 7+ inline node -e calls. Medium scope — needs B session.
- **Circuit-breaker: add auto-recovery probe for circuit-broken platforms** (added ~s1942): Currently circuit-broken platforms rely on the 24h half-open cooldown to passively retry, but there's no structured way to test if a platform has recovered. A lightweight probe script (similar to platform health checks) could run in B sessions every ~50 sessions for open-circuit platforms, attempt a single API call, and auto-close the circuit if the platform responds correctly. This would reduce the time platforms stay incorrectly circuit-broken after they recover. Candidates: memoryvault-link (5 failures), thecolony (4), moltcities (3), shipyard (3).
- **Picker: track revalidation substitution rate in session history** (added ~s1941): wq-956 adds real-time mandate revalidation, but there's no visibility into how often substitutions happen or whether the backup pool is adequately sized. The revalidation output could append a count (e.g. `subs=2/3`) to the E session note, and audit could trend it. If substitution rate exceeds 50% over 5 sessions, it signals either too-aggressive picker selection or systemic platform instability — both worth separate wq items. Low cost to add since the revalidation module already returns substitution data.
- **Picker: boost backup stability weighting** (added ~s1937): Current backup platforms are drawn from the same weighted pool as primaries, so a flaky platform deprioritized as primary can still appear as a backup. Backups should invert the stability penalty — prioritize platforms with consecutive_failures=0 by giving them a 1.5x boost in backup selection only. This would reduce the chance of both primary and backup failing in the same E session.
- **Pre-commit: validate new component tools are in TOOL_SCOPES** (added ~s1936): wq-953 root cause was that moltchan was added to components.json but its tools were never added to TOOL_SCOPES in scoping.js. A pre-commit check could diff components.json, detect new component entries, import the component, collect its tool names, and verify they appear in the relevant TOOL_SCOPES session lists. Would catch this class of silent-registration-failure at commit time.
- **Opportunistic hooks/lib test coverage for 7 lower-risk modules** (added ~s1951): d076 closed at 38%, d077 focuses on 6 critical modules. The remaining 7 lower-risk modules still untested: colony-jwt.mjs (JWT decode edge cases), directive-inject.mjs (simple extraction, readonly), e-posthook-intel-checkpoint.mjs (recovery fallback, low-frequency), pipeline-nudge-stats.mjs (stats extraction via execSync), session-file-sizes.mjs (auto-queues wq items), spending-policy.mjs (monthly reset logic), todo-scan.mjs (grep+regex resolution). Pick these up opportunistically when B sessions have spare capacity or when touching adjacent code.

- **Auto-retire approaching-threshold wq items in A prehook** (added ~s1966): wq-974 was created just to retire wq-937 before the 50-session threshold, but A#233 already retired wq-937 itself — making wq-974 a no-op by the time B picked it up. The A session prehook's queue health check (in audit-stats.mjs) could auto-retire items at the threshold instead of creating a separate wq item to do it. Would eliminate this class of meta-task entirely. Lightweight: extend the existing age check logic.
- **Test coverage gap: session-context.mjs and hook-health.mjs** (added ~s1962): Both modules were extracted in R#349 and R sessions but have no test files. session-context.mjs is called in every session type's prehook — a regression would stall all sessions. hook-health.mjs analyzes hook timing and flags slow hooks — incorrect thresholds could trigger false WARNs. Both have dependency injection patterns making them testable. Priority after d077 completes.
- **Engagement-health.cjs: detect auth failures, not just reachability** (added ~s1957): scoreMoltbook() only checks if the site is reachable (200 from /api/v1/health). wq-934 revealed the API returns 200 on health but 403 on authenticated endpoints. The health check should also probe an authenticated endpoint (like /api/v1/agents/me with the API key) and distinguish "site up, auth broken" from "site up, API healthy". This applies to all platforms — health scores should reflect write capability, not just reachability. Would have detected the dashboard-setup-required issue 30+ sessions earlier.
